───────────────────────────────────────────────────────────────────────────────
1. RESUMEN DEL PROYECTO
───────────────────────────────────────────────────────────────────────────────

TuHora es una plataforma móvil que conecta tres roles:

  · CLIENTES         → reservan turnos con profesionales independientes.
  · PROFESIONALES    → barberos, esteticistas, dentistas, etc., que NO tienen
                       local propio y necesitan un espacio para atender.
  · PROPIETARIOS     → personas con espacios físicos (sillones, consultorios,
                       boxes) que alquilan por hora a profesionales.

La aplicación funciona como un "Airbnb de espacios + Uber de turnos":

  ▸ El cliente busca al profesional, no al lugar.
  ▸ El profesional alquila el espacio que mejor le quede según horario y zona.
  ▸ El propietario monetiza horas vacías de su local.
  ▸ La plataforma cobra una comisión de cada transacción (configurable por
    profesional en `commissionRate`).

Diferenciadores frente a la competencia (Booksy, Fresha):

  · Triangulación automática: el sistema cruza disponibilidad de cliente,
    profesional y espacio en una sola consulta y devuelve los slots libres.
  · Validación de identidad (KYC) obligatoria para profesionales.
  · Gamificación: puntos, niveles (Bronce/Plata/Oro/Platino/Diamante), ruleta.
  · Pagos con split automático (Mercado Pago) — pendiente de implementar.

Decisiones clave:

  · Monorepo con dos carpetas: /frontend y /backend (independientes pero en
    el mismo repositorio). Permite compartir tipos en el futuro y tener un
    solo historial git.
  · REST en vez de GraphQL → más simple, mejor cacheo HTTP, suficiente para
    la complejidad actual. Si en el futuro las consultas anidadas crecen,
    se puede añadir GraphQL como capa adicional.
  · PostgreSQL en lugar de MongoDB → datos altamente relacionales (un turno
    referencia cliente, profesional, espacio, servicio, sesión, etc.).
    Las foreign keys e integridad referencial son críticas en este negocio.
  · Prisma como ORM → tipado fuerte (TypeScript), migraciones declarativas,
    y studio web gratis para ver datos en desarrollo.
  · Docker Compose para Postgres → cualquier integrante del equipo levanta
    la base en 1 comando, sin instalar Postgres en su máquina.

  ▸ Vite                        Bundler ultra rápido (sustituye a Webpack/CRA).

  ▸ TypeScript                  Tipado estático → menos bugs en runtime y

  ▸ Tailwind CSS v4             CSS utility-first. Permite estilar sin escribir

  ▸ Radix UI                    Primitivas de componentes accesibles (modales,
                                tooltips, switches) sin opinión visual.
  ▸ Shadcn UI                   Sistema de diseño basado en Radix + Tailwind,
                                con componentes copiables (no es una librería npm).
  ▸ Lucide React                Iconos SVG (Calendar, Home, Clock, etc.).
                                Ligero y consistente visualmente.
  ▸ sonner                      Notificaciones tipo toast (preferido por el
                                equipo, ver CLAUDE.md).

¿Por qué esta combinación y no Next.js o React Native directo?
  Estamos en fase de prototipo visual y validación de UX. Un proyecto Vite
  puro es más liviano, sin SSR ni rutas server-side innecesarias. Cuando
  llegue el momento, el código React se migra casi 1:1 a React Native.

4. BASE DE DATOS — modelado y por qué de cada tabla
───────────────────────────────────────────────────────────────────────────────

4.1 FILOSOFÍA DEL MODELO
─────────────────────────────────────

  · Una tabla `users` central con tres tablas de perfiles 1:1:
      client_profiles · professional_profiles · owner_profiles
    Esto permite que un mismo email pueda jugar varios roles si en el futuro
    se decide habilitarlo.
  · IDs UUID en todas las tablas (no enteros autoincrementales) → más seguro
    para exponer en URLs públicas y evita race conditions en sharding.
  · Soft-delete vía campos `status` (ACTIVE/INACTIVE/DELETED) en vez de borrar
    filas → mantiene historial para auditoría y reportes.
  · Snapshots en `appointments` (precio, comisión, datos del pro al momento de
    reservar) → si un servicio cambia de precio mañana, el turno de hoy
    conserva la info correcta.
  · Tablas de cache (caches/snapshots) opcionales para acelerar búsquedas
    de "espacios disponibles ahora".

4.3 ALGORITMO DE TRIANGULACIÓN
─────────────────────────────────────

El corazón funcional del producto. Ubicación: bookings.service.ts → checkAvailability().

ENTRADA:
  - professionalId
  - spaceId  (opcional, si el cliente quiere lugar específico)
  - date     (YYYY-MM-DD)
  - duration (minutos del servicio)

PROCESO:
  1. Lee professional_availabilities filtrando por dayOfWeek y obtiene
     ventana horaria del pro (ej. 09:00–18:00).
  2. Lee appointments existentes del pro ese día → conflictos.
  3. Si spaceId: lee space_availabilities + appointments del espacio →
     intersecta ventana horaria con la del pro (maxTime/minTime helpers).
  4. Genera slots de 30 minutos dentro de la ventana intersectada.
  5. Filtra los slots que se solapan con appointments existentes.

SALIDA:
  Array de strings ["09:00", "09:30", "10:00", ...]

Por qué 30 minutos:
  La mayoría de los servicios son múltiplos de 30 (corte 30 min, corte+barba
  45 min, fade 40 min). 15 minutos sería más granular pero generaría demasiados
  slots y la UI se ve recargada.

5. BACKEND — qué se usa, por qué, y cómo
───────────────────────────────────────────────────────────────────────────────

5.1 STACK Y POR QUÉ CADA PIEZA
─────────────────────────────────────

  ▸ NestJS 10                  Framework Node.js inspirado en Angular.
                                Estructura modular obligatoria → escala bien
                                en proyectos grandes y con equipos.
                                Decoradores estilo Spring/Angular para rutas.
  ▸ TypeScript                  Tipado fuerte en backend también.
  ▸ Prisma 5                    ORM con migraciones declarativas y cliente
                                tipado generado a partir de schema.prisma.
                                `prisma studio` da una UI web para ver datos.
  ▸ PostgreSQL 16               Base relacional probada, soporte JSON, índices
                                avanzados, FTS para búsquedas.
  ▸ Passport + JWT              Estándar de la industria para autenticación
                                stateless. Tokens cortos (access 15 min) +
                                refresh tokens largos (7 días).
  ▸ bcrypt                      Hashing de passwords (12 rounds).
  ▸ class-validator             Decoradores para validar DTOs en runtime.
                                Ej. @IsEmail() @MinLength(8) @IsUUID().
  ▸ class-transformer           Convierte payloads JSON a instancias de DTO.
  ▸ @nestjs/swagger             Documentación OpenAPI auto-generada en /docs.
  ▸ @nestjs/throttler           Rate limiting (anti brute-force).
  ▸ Docker Compose              Levanta Postgres en un comando.

¿Por qué NestJS y no Express puro?
  Express es minimalista pero te obliga a inventar tu propia estructura.
  En un equipo de 3 personas con poca experiencia compartida en backend,
  NestJS impone una estructura clara (módulos, controllers, services, DTOs)
  y minimiza las "decisiones de arquitectura" que normalmente generan
  inconsistencias entre miembros del equipo.

¿Por qué Prisma y no TypeORM o Sequelize?
  Prisma es declarativo: definimos el modelo en schema.prisma y el cliente
  TS se genera solo. Las migraciones son archivos SQL versionados en git.
  TypeORM tiene problemas históricos con migraciones automáticas en proyectos
  grandes. Sequelize es más viejo y menos tipado.

5.3 MÓDULO AUTH
─────────────────────────────────────

ENDPOINTS:
  POST /api/v1/auth/register      Crea user + perfil según rol elegido.
  POST /api/v1/auth/login         Recibe email/password → access + refresh.
  POST /api/v1/auth/refresh       Recibe refresh token → nuevo access token
                                   (rotación: el refresh viejo queda invalidado).
  POST /api/v1/auth/logout        Cierra sesión (borra el refresh de DB).

LÓGICA CLAVE EN auth.service.ts:
  · bcrypt.hash(password, 12)              → 12 rounds = ~250ms (balance OK).
  · MAX_FAILED_ATTEMPTS = 5                → al 5to login fallido, lockout.
  · LOCKOUT_DURATION = 15 minutos          → setea users.lockoutUntil.
  · Tokens:
      access  → 15 min (firmado con JWT_SECRET)
      refresh → 7 días (almacenado hasheado en auth_sessions)
  · Rotación de refresh: cada uso de /refresh invalida el refresh anterior
    y emite uno nuevo → si un atacante roba un refresh y la víctima lo usa,
    el atacante queda fuera (detectable como reutilización).

ESTRATEGIAS PASSPORT:
  · LocalStrategy: valida email/password → habilita el endpoint /login.
  · JwtStrategy:   extrae el JWT del header Authorization, lo verifica y
                    pone `req.user` con el usuario.

GUARDS:
  · JwtAuthGuard:   global por defecto (excepto rutas con @Public()).
  · LocalAuthGuard: protege /login.
  · RolesGuard:     verifica que el rol del JWT coincida con el @Roles() de
                     la ruta.

5.5 MÓDULO SPACES
─────────────────────────────────────

ENDPOINTS:
  GET    /api/v1/spaces                       Buscar espacios disponibles
                                                (filtros: ciudad, categoría,
                                                fecha, duración).
  POST   /api/v1/spaces                       [OWNER] Crear espacio.
  GET    /api/v1/spaces/my                    [OWNER] Mis espacios.
  GET    /api/v1/spaces/:id                   Detalle de un espacio.
  PATCH  /api/v1/spaces/:id                   [OWNER] Actualizar.
  DELETE /api/v1/spaces/:id                   [OWNER] Soft delete.
  PUT    /api/v1/spaces/:id/availability      [OWNER] Configurar horarios.
  GET    /api/v1/spaces/:id/slots             Slots libres (fecha + duración).

LÓGICA EN spaces.service.ts:
  · assertOwns(spaceId, userId) → función guard que tira ForbiddenException
    si el espacio no pertenece al user → previene IDOR (Insecure Direct
    Object Reference).
  · computeSlots(space, date, duration) → lee disponibilidad + appointments
    ocupados → genera lista de slots disponibles.


5.6 MÓDULO BOOKINGS
─────────────────────────────────────

Es el módulo más grande y crítico. Maneja dos flujos:

A) TURNOS (cliente reserva con profesional):
  GET    /api/v1/bookings/availability                   ⭐ Triangulación.
  POST   /api/v1/bookings/appointments                   [CLIENT] Reservar.
  GET    /api/v1/bookings/appointments/me                [CLIENT] Mis turnos.
  GET    /api/v1/bookings/appointments/professional      [PRO]    Turnos del pro.
  PATCH  /api/v1/bookings/appointments/:id/cancel        Cancelar.
  PATCH  /api/v1/bookings/appointments/:id/complete      [PRO] Marcar como hecho.

B) SOLICITUDES DE ESPACIO (profesional pide al propietario):
  POST   /api/v1/bookings/space-requests                 [PRO]    Solicitar.
  GET    /api/v1/bookings/space-requests/owner           [OWNER]  Ver pendientes.
  GET    /api/v1/bookings/space-requests/professional    [PRO]    Mis solicitudes.
  PATCH  /api/v1/bookings/space-requests/:id/respond     [OWNER] Aprobar/rechazar.

DECISIONES:
  · Snapshot de precio: al crear el appointment se copia el `priceSnapshot` del
    servicio en ese momento → si el pro sube el precio mañana, el turno de
    ayer mantiene el precio acordado.
  · Solo el cliente o el pro pueden cancelar (validado en service).
  · Auto-approve opcional: owner_profiles.autoApprove=true salta el flujo
    de espera y aprueba automáticamente las solicitudes (típico de owners
    que confían en el marketplace).


5.7 DECORADORES Y GUARDS COMUNES
─────────────────────────────────────

@Public()        Marca un endpoint como público (sin JWT necesario).
                  Implementación: SetMetadata('isPublic', true).
                  El JwtAuthGuard global lo lee y skipea la auth.

@Roles(roles)    Restringe un endpoint a roles específicos.
                  Ej: @Roles(UserRole.OWNER, UserRole.ADMIN).
                  Funciona junto con RolesGuard.

@CurrentUser()   Inyecta el usuario autenticado en el handler.
                  Reemplaza `@Req() req` + `req.user`.

RolesGuard       Lee la metadata @Roles() y compara contra req.user.role.

9. GLOSARIO
───────────────────────────────────────────────────────────────────────────────

  CTA              Call To Action — botón principal de una pantalla.
  DTO              Data Transfer Object — clase que valida un payload de API.
  HMR              Hot Module Reload — recarga de módulos sin perder estado.
  IDOR             Insecure Direct Object Reference — vulnerabilidad donde
                    cambiando un id en la URL accedés a datos de otro usuario.
  JWT              JSON Web Token — token firmado para autenticación stateless.
  KYC              Know Your Customer — validación de identidad.
  MVP              Minimum Viable Product.
  Slot             Bloque de tiempo disponible para reservar (ej. 30 min).
  Snapshot         Copia de un dato en un momento (ej. precio al reservar).
  SSR              Server-Side Rendering.
  Triangulación    Cruzar disponibilidades de cliente + pro + espacio.
